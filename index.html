Login Page / State Variables

02 State Variables — The Page's Memory

State variables are like the page's short-term memory. Everything that can CHANGE while the user is looking at the screen lives here.

🧠
Analogy — A Waiter's Notepad A waiter takes your order and writes it down. The notepad holds the "state" — table number, what was ordered, whether it's been served. In Flutter, our state variables are that notepad. When something changes, we update the notepad and the app redraws.

TextEditingControllers

Dart
login.dart
final TextEditingController _emailController    = TextEditingController();
final TextEditingController _passwordController = TextEditingController();

What is a TextEditingController? It's like a live wire attached to a text field. Whenever the user types something, the controller captures it. When we need to read what they typed, we do _emailController.text.

Why final? The controller object itself never changes (we don't replace it). The TEXT inside changes, but the controller stays the same object throughout the page's life.

💡
Practical example: User types john@company.com → _emailController.text now equals "john@company.com". It's that simple!

Boolean Flags

Dart
login.dart
bool _isLoading   = false; // Is the login button currently spinning?
bool _obscureText = true;  // Is password hidden? (shows ●●● when true)

_isLoading: When this is true, the button shows a spinner CircularProgressIndicator instead of the text "LOGIN". When the API call finishes, we set it back to false.

_obscureText: When true → password shows as ●●●. When user taps the eye icon, we flip it to false → password shows as plain text.

👁️
_obscureText is like a frosted glass window When true, the glass is frosted — you see the shape but not the content. Tapping the eye icon clears the glass. Setting it back to true frosts it again.

Form Key

Dart
login.dart
final _formKey = GlobalKey<FormState>();

Think of _formKey as the ID card of the form. Flutter needs this unique key to find and talk to your Form widget later. You give this key to the Form(key: _formKey) and then call _formKey.currentState!.validate() to trigger ALL validators at once.

⚠️
Why GlobalKey? Regular keys are local. A GlobalKey can be accessed from anywhere in the code, not just within its own widget tree. That's why it's called "Global."

Lockout State Variables

Dart
login.dart
int     _failedAttempts = 0;     // Counts wrong passwords (max 3)
bool    _loginDisabled  = false; // true = login button is locked
int     _timerCount     = 30;    // Countdown: starts at 30 seconds
Timer?  _timer;                  // The actual timer object (nullable)
String? _loginError;             // Error text shown in red (null = no error)

These 5 variables work together to implement the brute-force lockout feature:

  1. User enters wrong password → _failedAttempts++
  2. When _failedAttempts >= 3 → set _loginDisabled = true
  3. Start 30-second timer → _timerCount counts down each second
  4. When timer hits 0 → reset everything back to 0/false

The ? on Timer? and String? means they can be null. _timer starts as null because there's no timer running yet. _loginError is null when there's no error to show.

Before sending data to the server, we validate it locally on the device. This saves time (no wasted network call) and gives instant feedback to the user.

🚦

Login Page / Lockout Timer

04 The Lockout Timer — Brute Force Protection

After 3 failed login attempts, the app locks the login button for 30 seconds. This prevents automated bots from trying thousands of passwords. Here's exactly how it works.

Dart
login.dart — _startTimer()
void _startTimer() {
  _timerCount = 30; // Reset to 30 every time we call this

  // Timer.periodic fires a function every 1 second
  // It gives us a 'timer' object we can use to cancel it
  _timer = Timer.periodic(const Duration(seconds: 1), (timer) {
    setState(() {       // Redraw screen every second
      if (_timerCount > 0) {
        _timerCount--;   // Tick down: 30→29→28→...→1→0
      } else {
        _timer?.cancel();         // Stop the timer
        _loginDisabled  = false; // Unlock the button
        _failedAttempts = 0;     // Reset failure counter
        _loginError     = null;  // Clear error message
      }
    });
  });
}
⚠️
Why setState() inside the timer? The timer runs outside Flutter's normal drawing cycle. Wrapping updates in setState() tells Flutter "something changed, please redraw." Without it, the countdown number on screen would never update.
Dart
login.dart — dispose()
@override
void dispose() {
  _timer?.cancel(); // Cancel timer if page is closed mid-countdown
  super.dispose(); // ALWAYS call super.dispose() — cleans up the widget
}
🚰
dispose() is like turning off the tap when you leave If the user navigates away while the timer is running (30 second countdown), the timer would keep running in the background, wasting memory. dispose() is called automatically when the widget is removed from screen — it's our cleanup crew.
🚨
Memory Leak Warning: Forgetting to cancel timers in dispose() is a common bug. The timer keeps firing after the widget is gone, trying to call setState() on a dead widget — this crashes the app with "setState() called after dispose()."

Login Page / _login() Function

05 The _login() Function — Heart of the Page

This is the most important function. It runs when the user taps LOGIN. Let's break it into 5 clear chunks and understand each one deeply.

Guard Clauses — Stop Early if Something's Wrong

Dart
login.dart
Future<void> _login() async {

  // Guard 1: Is login currently locked? Stop immediately.
  if (_loginDisabled) {
    setState(() {
      _loginError = 'Login disabled. Try again in $_timerCount seconds';
    });
    return; // ← EXIT the function right here
  }

  // Guard 2: Did the user fill the form correctly?
  if (!_formKey.currentState!.validate()) {
    return; // ← Validator showed red errors, stop here
  }

  // If we reach this line, we're clear to proceed!
  setState(() {
    _isLoading  = true;
    _loginError = null; // Clear any old error
  });
}
💡
Guard Clauses are early return statements that prevent the function from running further when a condition isn't met. This keeps code clean — no deep nesting, no if-else-if-else spaghetti. Check for problems first, handle the happy path after.

The HTTP POST Request

Dart
login.dart

06 SharedPreferences — Saving to the Device

After a successful login, we save the user's ID and role on their phone. This is like a "Remember Me" feature — the app knows who you are even after restarting.

Future<void> saveIdAndRole(String id, String role) async { // Get the SharedPreferences instance // Like opening a key-value notepad on the phone SharedPreferences prefs = await SharedPreferences.getInstance(); // ① // Save as key-value pairs await prefs.setString('id', id); // ② key='id', value='5' await prefs.setString('role', role); // ③ key='role', value='Admin' print('Saved ID: $id'); print('Saved Role: $role'); }
①
getInstance()Opens the SharedPreferences storage on this device. await is needed because opening it takes a moment (reads from device storage). You get a prefs object to read/write with.
②
setString('id', id)Writes a String to the "id" key. On next app launch, we can read it back with prefs.getString('id'). The data persists until the app is uninstalled.
③
setString('role', role)Saves "Admin" or "Employee". Other screens can read this to decide what UI to show without having to ask the server again.

setInt(key, val)Save an IntegersetInt('loginCount', 3)
setBool(key, val)Save a BooleansetBool('darkMode', true)
remove(key)Delete a valueremove('role') on logout

09 ForgotPasswordPage — Two-API Flow

This page handles a 2-step flow: first check if the email exists, then send the OTP. Understanding how two sequential API calls work is key here.

📱 User
Types email → taps "Send Recovery Email"
① PHP
check_email.php → checks if email exists in DB → returns {"success": true/false}
② PHP
If email exists → otp.php → generates OTP, emails it → returns {"success": true, "otp": "123456"}
📱 Flutter
Navigates to OtpPage, passes email + otp
Dart
ForgotPasswordPage.dart
Future<void> _sendRecoveryEmail() async {
  if (!_formKey.currentState!.validate()) return;

  setState(() { _isLoading = true; _errorMessage = null; });

  try {
    // ── CALL 1: Does this email exist? ──
    final response = await http.post(
      Uri.parse('http://192.168.29.211/hr_api/check_email.php'),
      headers: {'Content-Type': 'application/json'}, // ← sends JSON
      body: jsonEncode({'email': _emailController.text}),
    );

    final responseData = jsonDecode(response.body);

    if (responseData['success'] == true) {

      // ── CALL 2: Only now send the OTP ──
      final otpResponse = await http.post(
        Uri.parse('http://192.168.29.211/hr_api/otp.php'),
        headers: {'Content-Type': 'application/json'},
        body: jsonEncode({'email': _emailController.text}),
      );
      final otpData = jsonDecode(otpResponse.body);

      setState(() { _isLoading = false; });

      if (otpData['success'] == true) {
        final String? otp = otpData['otp']?.toString(); // ← ?. safe access
        Navigator.push(context, MaterialPageRoute(
          builder: (context) => OtpPage(
            email: _emailController.text,
            primary: primary,
            otp: otp,
          ),
        ));
      }
    } else {
      setState(() {
        _isLoading    = false;
        _errorMessage = responseData['message'] ?? 'Email not found';
      });
    }
  } catch (e) {
    setState(() { _isLoading = false; _errorMessage = 'An error occurred.'; });
  }
}

  body: jsonEncode({'email': email}),
);
// PHP reads: file_get_contents('php://input')

Both approaches work. The PHP side just needs to know which format to expect. Form data is simpler; JSON is more flexible and better for complex payloads.

The ?? (Null Coalescing) Operator

Dart
// The ?? operator means: "use this IF the left side is null"
_errorMessage = responseData['message'] ?? 'Email not found';

// Same as writing:
if (responseData['message'] != null) {
  _errorMessage = responseData['message'];
} else {
  _errorMessage = 'Email not found';
}

// The ?. (null-safe access) operator:
final String? otp = otpData['otp']?.toString();
// If otpData['otp'] is null, don't crash — just return null
// Without ?. it would crash: "Cannot call toString() on null"
🔀
?? is like a "backup plan" "Use my first choice. If it's null (doesn't exist), use the backup." Like saying: "I'll take the window seat, or if that's taken, any aisle seat."

10 OTP Page — Dual Verification Strategy

The OTP page is clever — it can verify the code in two ways. This is a great pattern to understand.

Future<void> _verifyOtp(String enteredOtp) async { if (enteredOtp.length != 6) { // Basic length check setState(() => _errorMessage = 'Please enter all 6 digits'); return; } setState(() { _isVerifying = true; _errorMessage = null; }); try { // ── Strategy 1: Client-side check ── // ① if (widget.otp != null && widget.otp == enteredOtp) { setState(() => _isVerifying = false); Navigator.pushReplacement(context, MaterialPageRoute( builder: (context) => ResetPasswordPage(email: widget.email), )); return; // ← Skip the server call if client check passes } // ── Strategy 2: Server-side check ── // ② final response = await http.post( Uri.parse('...verify_otp.php'), body: jsonEncode({'email': widget.email, 'otp': enteredOtp}), ); final responseData = jsonDecode(response.body); setState(() => _isVerifying = false); if (responseData['success'] == true) { Navigator.pushReplacement(context, MaterialPageRoute( builder: (context) => ResetPasswordPage(email: widget.email), )); } } catch(e) { /* handle error */ } }
①
Client-side check (FAST)If the PHP otp.php returned the OTP in its response (stored in widget.otp), we can verify locally without another network call. Faster UX!
②
Server-side check (FALLBACK)If widget.otp is null (server didn't return OTP in response), we send it to verify_otp.php to check. More secure — the OTP lives only on the server.
⚠️
Security Note on Client-Side OTP: When the OTP is passed to the Flutter app in the API response (widget.otp), it can be intercepted by someone with network access. For maximum security, always verify on the server (Strategy 2 only). Strategy 1 is a development shortcut.

    return 'Password must contain at least one special character';

  return null; // All checks passed!

Backend PHP / login.php

12 PHP Backend — How login.php Works

The PHP backend receives what Flutter sends and talks to the database. Here's the complete annotated implementation you'd write for SMART-HR.

PHP
login.php
// Allow requests from any origin (Flutter app)
header("Access-Control-Allow-Origin: *");
header("Access-Control-Allow-Methods: POST, OPTIONS");
header("Content-Type: application/json");

// Only process POST requests
if ($_SERVER['REQUEST_METHOD'] !== 'POST') {
    echo json_encode(["success" => false, "message" => "POST required"]);
    exit;
}

// Read form data (Flutter login.dart sends as form data, not JSON)
$email    = trim($_POST['email']    ?? '');
$password = trim($_POST['password'] ?? '');

if (empty($email) || empty($password)) {
    echo json_encode(["success" => false, "message" => "Missing fields"]);
    exit;
}

include("dbconnection.php");
$con = dbconnection();

// Fetch user by email — prepared statement prevents SQL injection
$stmt = mysqli_prepare($con,
    "SELECT userid, password, role FROM users WHERE email = ?");
mysqli_stmt_bind_param($stmt, "s", $email);
mysqli_stmt_execute($stmt);
mysqli_stmt_bind_result($stmt, $userid, $hashedPassword, $role);
mysqli_stmt_fetch($stmt);
mysqli_stmt_close($stmt);

if (isset($userid)) {
    // password_verify checks entered password against the stored hash
    if (password_verify($password, $hashedPassword)) {
        echo json_encode([
            "success" => true,
            "userid"  => $userid,
            "role"    => $role
        ]);
    } else {
        echo json_encode(["success" => false, "message" => "Invalid password"]);
    }
} else {
    echo json_encode(["success" => false, "message" => "Email not found"]);
}
mysqli_close($con);

Why $_POST['email'] (not json_decode)?

The Flutter login page sends data as form data (no Content-Type header, body is a Map). PHP reads form data via $_POST['key']. The ForgotPassword pages send JSON, so those PHP files use json_decode(file_get_contents('php://input')).

Form data (no Content-Type)$_POST['key']
JSON (Content-Type: application/json)json_decode(file_get_contents('php://input'))

Password Hashing — Never Store Plain Passwords!

❌ WRONG — Plain Text
✅ CORRECT — Hashed
// Registration (NEVER DO THIS):
INSERT INTO users
  (password) VALUES ('mypassword')

// Login check (NEVER DO THIS):
WHERE password = 'mypassword'
// If DB is hacked, all passwords exposed!
// Registration (CORRECT):
$hash = password_hash($password, PASSWORD_DEFAULT);
INSERT INTO users (password) VALUES ($hash)

// Login check (CORRECT):

// Hash can't be reversed — safe even if DB is hacked!

13 otp.php — Generating & Sending OTP

PHP
otp.php
header("Access-Control-Allow-Origin: *");
header("Content-Type: application/json");

// Read JSON body (ForgotPasswordPage sends JSON)
$input = json_decode(file_get_contents('php://input'), true);
$email = $input['email'] ?? '';

if (empty($email)) {
    echo json_encode(["success" => false, "message" => "Email required"]);
    exit;
}

// Generate a random 6-digit OTP
// rand(100000, 999999) gives a 6-digit number
$otp = rand(100000, 999999);

// Save OTP to database with expiry (for server-side verify_otp.php)
include("dbconnection.php");
$con = dbconnection();

$stmt = mysqli_prepare($con,
    "INSERT INTO otp_tokens (email, otp, expires_at)
     VALUES (?, ?, DATE_ADD(NOW(), INTERVAL 10 MINUTE))
     ON DUPLICATE KEY UPDATE otp=?, expires_at=DATE_ADD(NOW(), INTERVAL 10 MINUTE)");
mysqli_stmt_bind_param($stmt, "sss", $email, $otp, $otp);
mysqli_stmt_execute($stmt);

// Send OTP via email using PHP mail()
$subject = "Your SMART-HR OTP Code";
$message = "Your OTP is: $otp. Valid for 10 minutes.";
$headers = "From: noreply@smarthr.com";

if (mail($email, $subject, $message, $headers)) {
    // Include otp in response for client-side verification (dev convenience)
    echo json_encode(["success" => true, "otp" => $otp]);
} else {
    echo json_encode(["success" => false, "message" => "Failed to send email"]);
}
mysqli_close($con);
🚨
Security Note: Returning "otp": $otp in the response means the OTP travels over the network. In production, remove this and only verify via verify_otp.php. The OTP should only arrive via email to the legitimate user's inbox.

14 reset_password.php — Updating the Password

PHP
reset_password.php
header("Access-Control-Allow-Origin: *");
header("Content-Type: application/json");

// Read JSON body
$input    = json_decode(file_get_contents('php://input'), true);
$email    = $input['email']    ?? '';

if (empty($email) || empty($password)) {
    echo json_encode(["success" => false, "message" => "Missing fields"]);
    exit;
}

include("dbconnection.php");
$con = dbconnection();

// Hash the new password before storing
// NEVER store plain text passwords!
$hashedPassword = password_hash($password, PASSWORD_DEFAULT);

// Update the user's password in the database
$stmt = mysqli_prepare($con,
    "UPDATE users SET password = ? WHERE email = ?");
mysqli_stmt_bind_param($stmt, "ss", $hashedPassword, $email);

if (mysqli_stmt_execute($stmt)) {
    // Clean up: delete the used OTP token
    $del = mysqli_prepare($con,
        "DELETE FROM otp_tokens WHERE email = ?");
    mysqli_stmt_bind_param($del, "s", $email);
    mysqli_stmt_execute($del);

    echo json_encode(["success" => true, "message" => "Password reset successfully"]);
} else {
    echo json_encode(["success" => false, "message" => "Failed to update password"]);
}
mysqli_close($con);
