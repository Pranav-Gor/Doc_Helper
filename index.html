import 'dart:io'
noob Gives you access to the File class. You need this whenever you work with files on the phone — like a photo the user picked from their gallery. Without this, Dart doesn't know what a "File" is.
advanced → dart:io is the Dart standard library for OS-level stuff: files, sockets, directories. It's not available in Flutter Web builds — use dart:html there instead.
import 'package: flutter/material.dart'
noob This one import gives you everything Flutter UI — Scaffold, Text, Container, Colors, Icons, every widget. You put this at the top of every single Flutter screen file.
advanced → material.dart re-exports cupertino, widgets, painting, rendering, gestures and more. It's an umbrella import.
import '...http.dart' as http
noob Brings in the HTTP package so you can send data to your PHP server. The as http part means you call it like http.post() instead of just post() — avoids name conflicts.
advanced → aliasing with 'as' is essential when two packages export the same name. http.Client also lets you reuse connections for multiple requests.
import 'dart:convert'
noob Gives you jsonEncode() and jsonDecode() — the two functions that convert Dart maps to JSON strings and back. You need this every time you talk to a server.
advanced → dart:convert also has Base64, Latin1, UTF8 codecs. jsonDecode returns dynamic — always cast to Map or List immediately.
import 'image_picker'

advanced → StatefulWidget is a two-class pattern by design: the Widget is immutable (rebuilt often), the State persists across rebuilds. This separation is intentional for Flutter's diffing algorithm.

Variable Explanations

int _selectedTab = 0
noob A number that remembers which tab the user is on. 0 = Hospital tab, 1 = Patient tab, 2 = Driver tab. When the user taps a tab, we call setState(() => _selectedTab = index) to update it and rebuild the UI.
advanced → Using an int index is simpler than an enum here, but for larger apps with many tabs, an enum (TabType.hospital) is safer — prevents invalid values like _selectedTab = 99.
GlobalKey<FormState>
noob A unique key that links your code to the Form widget. With it, you can call _hospitalFormKey.currentState!.validate() which checks all fields at once. Without it, you can't trigger validation from outside the form.
advanced → GlobalKey holds a reference to a widget's State across rebuilds. The ! (bang operator) asserts the state is non-null — safe here because the Form is always in the tree when validate() is called.
File? _hospitalPhoto
noob Holds the photo the user picks from their gallery. The ? after File means it can be null — it starts as null (no photo), then becomes a File object when the user picks one.
advanced → File? uses Dart's null safety. The compiler forces you to handle the null case. We check if (_hospitalPhoto == null) before uploading to avoid a null dereference crash.
bool _hasExistingDiseases = false
noob Controls the disease toggle on the Patient form. When the Switch is turned ON, this becomes true, and the disease text field appears. When OFF (false), the text field is hidden.
advanced → This is a classic conditional rendering pattern: if (_hasExistingDiseases) ...[widget1, widget2]. The spread operator ... unpacks the list directly into the Column's children list.
String? _driverLatitude
noob Stores the GPS latitude as text. Starts as null (no location captured yet). When the user taps "Capture My Location", this gets filled. The GPS button turns green when it's not null.
advanced → Storing GPS as String for easy multipart form sending. In a real app, store as double for calculations (distance, sorting nearby drivers). Convert with double.parse(_driverLatitude!).

Future<void> _pickImage(Function(File) setter) async {
  final XFile? picked = await _picker.pickImage(
    source: ImageSource.gallery,
    imageQuality: 80,
  );
  if (picked != null) {
    setState(() => setter(File(picked.path)));
  }
}

_pickImage((f) => _ambulancePhoto = f);

Line by Line

Function(File) setter
noob This parameter is itself a function! You pass in a mini function that tells _pickImage "where to store the result". Instead of writing 3 separate pick functions for each photo, one function works for all by accepting a setter.
advanced → This is a callback / higher-order function pattern. Function(File) is the type signature — any function that takes a File parameter. Equivalent to passing a lambda as an argument.
XFile? picked
noob XFile is what image_picker returns — a cross-platform file object. The ? means it can be null — it will be null if the user opens the gallery then cancels without picking anything.
advanced → XFile is from the cross_file package (a dependency of image_picker). It abstracts over dart:io File and dart:html for web compatibility. We convert it to a dart:io File via File(picked.path).
imageQuality: 80
noob Compresses the image to 80% quality before storing it. A hospital photo at full quality might be 5–10MB. Compressed to 80% it's typically under 1MB — much faster to upload and easier on your server.
advanced → imageQuality only works for JPEG compression. PNG files are losslessly converted to JPEG first. For medical reports, consider using a higher quality (90-95) to preserve text readability in documents.
if (picked != null)
noob Only runs the code inside if the user actually picked a photo. If they opened the gallery and pressed the back button, picked is null and we do nothing — we don't clear any previously selected photo.
advanced → This is a null guard. Without it, File(null.path) would crash. Dart's null safety would actually flag this at compile time, but explicit null checks keep the logic readable.
setState(() => setter(f))
noob Calls the setter function (which assigns the File to the right variable), wrapped inside setState so Flutter knows to redraw the upload box — showing the image preview instead of the "upload" placeholder.
advanced → Without setState, the File variable updates in memory but the UI stays frozen. setState schedules a build() call. The lambda (f) => _hospitalPhoto = f is a closure that captures the outer variable.

Widget _buildField({
  required TextEditingController controller,
  required String hint,
  required IconData icon,
  bool obscure = false,
  bool isPassword = false,
  VoidCallback? onToggleObscure,
  TextInputType keyboard = TextInputType.text,
  String? Function(String?)? validator,
  int maxLines = 1,
}) {
  return Container(
    decoration: BoxDecoration(
      color: Colors.white,
      borderRadius: BorderRadius.circular(50),
      boxShadow: const [
        BoxShadow(color: Colors.black12, blurRadius: 10,
                  spreadRadius: 2, offset: Offset(3,3)),
      ],
    ),
    child: TextFormField(
      controller: controller,
      obscureText: obscure,
      keyboardType: keyboard,
      maxLines: maxLines,
      validator: validator ?? (v) =>
          v == null || v.isEmpty ? 'This field is required' : null,
      decoration: InputDecoration(
        border: InputBorder.none,
        hintText: hint,
        prefixIcon: Icon(icon, color: Colors.redAccent),
        suffixIcon: isPassword ? IconButton(
          icon: Icon(obscure ? Icons.visibility_off : Icons.visibility),
          onPressed: onToggleObscure,
        ) : null,
      ),
    ),
  );
}

Key Concepts

required vs optional params
noob required means you MUST pass this value when calling _buildField(). If you forget it, Dart gives a compile error. Without required, the parameter is optional and needs a default value like bool obscure = false.
advanced → Named parameters with required are enforced at compile time (not runtime). This is safer than positional parameters for functions with many arguments.
validator ?? (v) =>
noob The ?? means "if validator is null, use this default function instead". So if you don't pass a custom validator, every field automatically validates as required. If you pass your own validator (like email check), it uses yours.
advanced → validator is typed as String? Function(String?)? — a nullable function that takes a nullable String and returns a nullable String. Returning null = valid. Returning a string = error message.
InputBorder.none
noob Removes the default underline that TextFormField shows. We remove it because our Container already acts as the visual border with its shadow. Keeping the default border would look double-styled.
advanced → You must set BOTH border: and enabledBorder: to InputBorder.none to fully remove all borders. Setting only border: still shows borders in some states.

Widget _buildSubmitButton(String label, VoidCallback onTap) {
  return GestureDetector(
    onTap: _isLoading ? null : onTap,   // null disables it
    child: Container(
      height: 60, width: double.infinity,
      decoration: BoxDecoration(
        color: _isLoading ? Colors.grey : Colors.redAccent,
        borderRadius: BorderRadius.circular(30),
      ),
      child: Center(
        child: _isLoading
            ? const CircularProgressIndicator(color: Colors.white, strokeWidth: 2)
            : Text(label, style: const TextStyle(color: Colors.white, letterSpacing: 3)),
      ),
    ),
  );
}

Key Concepts

ClipRRect
noob Clips its child widget into a rounded rectangle shape. Used here so the image preview doesn't overflow outside the rounded Container corners — without ClipRRect, the image corners would be sharp even though the box is rounded.
advanced → ClipRRect uses a RRect (rounded rectangle) clip path. It's a painting layer operation. For circular clips, use ClipOval instead. Both add a compositing layer which has a small performance cost.
file != null ? ... : ...
noob A ternary — one line if/else. If file is not null (photo picked), show Image.file(). If file IS null (nothing picked yet), show the upload icon + label. This makes the box show a live preview after picking.
advanced → Image.file() reads directly from the filesystem path. It caches the image in the widget tree. For large images, consider using Image.file with cacheWidth to limit memory usage.
onTap: _isLoading ? null : onTap
noob If we're currently loading (waiting for server response), set onTap to null — this disables the GestureDetector, preventing the user from tapping submit twice while an upload is in progress.
advanced → GestureDetector with onTap: null still renders normally (no grey-out). For visual feedback, we also set the Container color to grey when _isLoading is true.

Key Concepts

Form(key: _hospitalFormKey)
noob Form is an invisible wrapper widget. By itself it does nothing visually. But paired with a GlobalKey, it lets you call .validate() to check every TextFormField inside it at once — great for catching empty or invalid inputs.

validator: return null
noob In Flutter validators: returning null means the field is valid. Returning a String means it's invalid — and that string becomes the red error text shown below the field. This is the opposite of what beginners expect!

onTap: () => _pickImage((f) => _hospitalPhoto = f)
noob When the upload box is tapped: calls _pickImage, passing a function that says "store the picked file into _hospitalPhoto". This uses the reusable picker with a specific destination each time.
advanced → (f) => _hospitalPhoto = f is a closure. It captures _hospitalPhoto from the outer scope. Inside setState() the assignment triggers a rebuild showing the image preview.

Future<void> _submitHospital() async {
  if (!_hospitalFormKey.currentState!.validate()) return;
  if (_hospitalPhoto == null) {
    _showMessage("Please upload hospital photo");
    return;
  }

  setState(() => _isLoading = true);

  try {
    final request = http.MultipartRequest(
      'POST',
      Uri.parse('http://10.0.2.2/ambulance_api/register_hospital.php'),
    );

    // ── Text fields ──
    request.fields['hospital_name']     = _hospitalNameController.text.trim();
    request.fields['hospital_location'] = _hospitalLocationController.text.trim();
    request.fields['reg_number']        = _hospitalRegNoController.text.trim();
    request.fields['email']             = _hospitalEmailController.text.trim();
    request.fields['password']          = _hospitalPasswordController.text.trim();

    // ── File attachment ──
    request.files.add(
      await http.MultipartFile.fromPath('hospital_photo', _hospitalPhoto!.path),
    );

    final response = await request.send();
    final body     = await response.stream.bytesToString();
    final data     = jsonDecode(body);

    if (response.statusCode == 200 && data['success'] == true) {
      _showMessage("Registered successfully!", isError: false);
    } else {
      _showMessage(data['message'] ?? "Registration failed");
    }
  } catch (e) {
    _showMessage("Network error: $e");
  } finally {
    setState(() => _isLoading = false);
  }
}

Why MultipartRequest, not http.post()?

💡
http.post() sends only text (JSON or form fields). http.MultipartRequest sends a mixed payload — text fields AND binary file data together. This is how HTML form file uploads work under the hood (multipart/form-data encoding).
request.fields['key'] = value
noob Adds a text value to the request. Like filling out a form on paper — each key is the field name, each value is what you typed. PHP reads these with $_POST['key'].
advanced → fields in MultipartRequest are sent as form-data parts with Content-Disposition: form-data. PHP's $_POST and $_FILES are populated from these parts.
MultipartFile.fromPath('hospital_photo', path)
noob Attaches a file to the request. The first argument 'hospital_photo' is the key — PHP uses $_FILES['hospital_photo'] to access it. The second argument is the file's location on the phone.
advanced → fromPath() reads the file asynchronously (hence await). It auto-detects MIME type from the file extension. You can also pass contentType: MediaType('image','jpeg') explicitly.
response.stream.bytesToString()
noob MultipartRequest returns a StreamedResponse, not a regular Response. You can't read it directly — you must call .stream.bytesToString() to convert the stream into a plain string that jsonDecode can read.
advanced → StreamedResponse streams the body in chunks (efficient for large responses). bytesToString() collects all chunks and joins them. For large file downloads, process the stream chunk-by-chunk instead.

Key Concepts

? '1' : '0'
noob PHP and databases don't understand Dart's true/false directly. We convert the boolean to the string '1' (true) or '0' (false). In PHP, if ($data['has_diseases'] == '1') checks this correctly. MySQL stores it as TINYINT(1).
advanced → Alternatively send as JSON with jsonEncode and the bool transfers natively. But MultipartRequest fields are always strings, so this conversion is required for multipart submissions.

noob Medical report is optional — the patient might not have one. This check means: only add the file to the request if the user actually uploaded something. If they didn't, we send the form without any file — PHP handles this case.
advanced → PHP side: check with isset($_FILES['medical_report']) && $_FILES['medical_report']['error'] == 0 before processing. Don't require a file that was declared optional on the Flutter side.

Key Concepts

List.generate(n, (i) => ...)
noob Creates a List of n items. For each item, runs the function with the index (0,1,2...). Instead of writing 3 identical tab widgets by hand, we write it once and generate 3 from the tabs list. Much cleaner.

Key Concepts

AnimatedSwitcher
noob When its child widget changes, AnimatedSwitcher fades out the old one and fades in the new one. Without it, switching tabs would be an instant jarring swap. It automatically detects when child changes by comparing widget keys.
advanced → AnimatedSwitcher uses crossfade by default (FadeTransition). For slide transitions, provide a custom transitionBuilder. Important: for proper detection, give each child a unique Key when they're the same widget type.
dispose()
noob Called automatically by Flutter when the user leaves this screen. TextEditingControllers hold memory — if you don't dispose them, they keep running in the background even after the screen is gone. This causes memory leaks.
advanced → dispose() is part of the State lifecycle: initState() → build() → setState() → dispose(). Always dispose: TextEditingControllers, AnimationControllers, StreamSubscriptions, FocusNodes, and ScrollControllers.

noob Must be called at the end of your dispose(). It calls Flutter's own cleanup code in the parent State class. If you forget it, or call it before your own cleanup, Flutter throws an error.
advanced → Contrast with initState() where super.initState() must be called FIRST. The asymmetry: in init you set up before parent, in dispose you clean up before parent tears down.

Line by Line

$_POST['key']
noob PHP automatically populates $_POST with all text fields from a multipart form request. Flutter sent request.fields['hospital_name'] = 'City Hospital' → PHP reads it as $_POST['hospital_name'].
advanced → Note: when Flutter sends JSON body (jsonEncode), PHP uses php://input, not $_POST. $_POST is only populated for multipart/form-data or application/x-www-form-urlencoded content types.
password_hash(..., PASSWORD_BCRYPT)
noob NEVER store passwords as plain text. password_hash() scrambles the password into a one-way hash — no one can un-scramble it, not even you. To check a login: use password_verify($input, $storedHash).
advanced → PASSWORD_BCRYPT uses cost factor 10 by default (2^10 = 1024 iterations). Increase cost for better security (at the cost of speed). PASSWORD_DEFAULT is preferred as it updates to stronger algorithms automatically.
$_FILES['hospital_photo']['error'] === 0
noob PHP gives every uploaded file an error code. 0 means success (no error). Always check this before processing the file. Other codes: 1=file too large, 2=form MAX_FILE_SIZE exceeded, 4=no file uploaded.
advanced → $_FILES['key']['tmp_name'] is where PHP temporarily stores the upload. It gets deleted after the request ends unless you move it with move_uploaded_file().
uniqid('hosp_', true)
noob Generates a unique filename like hosp_6631ab2c3f4e21.23. Never use the original filename from the user — it could be malicious (e.g. ../../config.php) or collide with existing files.
advanced → uniqid is based on microtime, not cryptographically random. For better uniqueness use bin2hex(random_bytes(16)). For production file storage, consider cloud storage (S3, Cloudinary) instead of local filesystem.
PDO + prepare() + execute()
noob PDO is PHP's safe database layer. prepare() creates the query with :placeholders. execute() fills in the values safely. This prevents SQL injection — a hacker can't break out of the placeholders.
advanced → PDO prepared statements send the query structure and data separately to the DB server. The DB handles escaping. This is fundamentally safer than string concatenation (even with mysqli_real_escape_string).
